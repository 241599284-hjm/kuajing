## 项目定位

自营跨境茶具/工艺品独立站，同一源码按客户独立部署，不做运行时多租户 SaaS，不增加 `tenant-service` 或 `tenant_id`。

核心交易链路：
后台商品与库存 → 前台浏览/购物车 → Guest Checkout → 本地订单与库存预留 → PayPal → 验签 Webhook → 支付确认与库存扣减 → 后台履约/退款。

## 工程不变量

- 前台功能必须有后台、真实接口和持久化支撑；不得以静态兜底、假保存或假成功替代生产闭环。
- 商品、分类、地域、详情、SEO、媒体及业务配置必须后台可维护；买家侧中英双语，后台默认中文。
- 一个业务模块一个微服务边界，前后台不直连数据库；金额、库存、数量用整数最小单位，时间存 UTC。
- 支付、物流、税费、汇率、风控走 Provider 适配层；禁止把供应商逻辑硬编码进订单主流程。
- 写接口、上传、下单、退款和库存操作必须幂等；Saga/TCC 补偿真实反向改数据，失败持久化重试，耗尽进入 DLQ。
- DLQ 和 failed reconciliation 必须有后台查看、重试、作废与审计入口。
- 同步 API 失败统一返回业务错误码、明确文案和 correlation ID。Provider Webhook 是异步特例：完成验签并持久写入 durable inbox 后返回 `202`；验签或协议错误按 Provider 要求返回失败状态，禁止伪造 `200/202` 成功。
- 后台敏感写操作记录操作人、时间、IP、请求 ID 和变更；危险操作统一二次确认。
- 列表分页；`PUT` 表示全量替换，`PATCH` 表示增量更新。
- Catalog 高频读取使用维度化 Redis 缓存；数据库提交后失效相关缓存，静态兜底不得写缓存。
- 邮件统一走 `notification-service` 的模板、账号池和发送日志；评价必须绑定已购订单/订单行，审核后才展示。
- PC、平板、手机均需验收；移动端触控区域不小于 44px。
- 视觉遵循 `docs/premium-minimal-visual-system.md`、`docs/module-visual-templates.md` 和后台设计规范，优先复用公共组件。
- 当前使用 Docker Compose；交易闭环稳定后再评估 K8s，生产 PostgreSQL/Redis 不放入 K8s。

## URL、HTTPS 与媒体

此节是唯一权威规则，其他章节不得重复改写：

- 浏览器 API、认证和媒体地址来自环境变量或同源反向代理。开发可用 `http://localhost`；生产必须 HTTPS，业务组件禁止写死线上域名、IP或协议。
- Docker 内部可用 `http://service-name:port`，但不得返回浏览器或写入业务数据。
- 站内静态资源保存 `/assets/*`、`/static/*` 等相对路径；上传媒体保存 `/media/public/{objectKey}`、`objectKey` 和轻量元数据。
- 禁止持久化测试 IP、正式域名、CDN 完整地址、`http://127.0.0.1/*` 或其他完整站内 HTTP URL。
- 第三方资源保存显式 `https://`；输入为 `//` 时先规范化为 HTTPS。无法提供 HTTPS 的资源不得进入生产页面。
- 富文本、邮件模板、评价、商品、分类和地域媒体在写接口与数据库约束两层拒绝 `http://`。
- `/media/public/*` 由前后台同源代理到 media-service；HTTPS/CDN 切换只改环境与边缘配置，不批量改业务库。
- 上传、绑定、解绑、响应式 WebP/视频派生对象清理及不确定 5xx durable reconciliation 必须闭环。

## 交互与后台

- 同类业务列表唯一规范：`docs/list-detail-interaction-standard.md`。列表固定单行摘要，文本截断；仅操作列详情按钮触发；详情独立接口；页面只有一个 Portal 弹窗；关闭清空数据，再次打开重新请求。
- 后台页面形态：`docs/admin-list-and-config-page-standard.md`。商品和分类在列表内新增、详情、修改；全局单例使用配置表单；多记录资源使用列表规范。
- 所有第三方接口或关键依赖配置都要有真实“测试连通性”按钮，必须穿透实际网关、服务和依赖，不得模拟成功。
- 未单独描述的页面沿用统一 Layout、加载、空状态、错误、分页、权限、审计和响应式规则。
- 首页保存草稿不影响买家页；发布、删除及其他危险操作适用统一二次确认规则。
