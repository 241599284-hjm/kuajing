# 业务列表与详情弹窗强制规范

适用于前台和后台的订单、商品、客户、售后、退款、库存流水及其他同类业务列表。新页面必须直接遵守；修改旧页面时必须同步迁移，不得继续扩散旧式卡片详情或行点击交互。

## 列表

- 单条业务数据固定渲染为单独一行，只显示识别、状态、金额、时间等摘要字段。
- 文本保持单行并使用省略号截断；金额统一按币种显示两位小数；图片必须有加载失败兜底。
- 使用业务唯一 ID 或稳定 slug 作为 React key 和详情请求参数，禁止使用数组下标。
- 点击行、单元格和空白区域不得打开详情；仅操作列的“详情”按钮可以触发。
- PC 表格固定行高，操作列固定右侧；移动端表格在自身容器内滚动，不得撑宽整页。
- 大数据列表必须使用服务端分页；单页达到性能阈值时使用虚拟滚动，不得一次挂载无界 DOM。
- 无详情查看权限时隐藏详情按钮，不能只在弹窗内做视觉阻止。

## 详情请求

- 详情必须通过独立接口按业务唯一 ID 重新请求，禁止复用列表摘要对象渲染完整详情。
- 页面只能维护一个详情弹窗实例；弹窗与列表平级，通过 Portal 挂载页面根节点，禁止写在列表循环内。
- 打开时先清空旧详情，再进入 loading；加载期间禁用全部详情按钮，防止重复请求和数据串扰。
- 新请求必须中止旧请求或忽略过期响应；多标签页面分别维护选中 ID 和请求状态。
- 接口空响应、404、403、数据已删除和依赖失败必须显示明确空态或错误态，禁止白屏和伪造成功。

## 弹窗

- 弹窗居中，最大高度不超过可视区，标题区固定，内容区独立滚动。
- 点击遮罩、关闭按钮或按 ESC 均可关闭；弹窗阻止事件穿透并锁定底层页面滚动。
- 移动端四周至少保留 16px 安全边距，主要操作按钮触摸尺寸不得小于 44px。
- 关闭时必须中止请求，并清空选中 ID、详情数据、loading、错误和业务操作状态；再次打开必须重新请求。
- 内嵌确认弹窗使用更高层级，危险写操作继续执行二次确认。

## 验收

- 自动化测试至少覆盖：打开清旧数据、过期响应忽略、关闭全量清理。
- Playwright 至少覆盖：行空白无触发、按钮唯一触发、加载锁、重新请求、ESC/遮罩关闭、底层滚动锁和桌面/移动端无整页溢出。
